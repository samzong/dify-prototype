# 05. 产品经理产品意识矩阵

## Dataset 产品意识

Dataset 产品经理不能只看“上传文档”和“检索参数”。这个模块的真实链路有六层：

1. 数据进入系统。
2. 数据被解析、清洗、分段，变成结构化资产。
3. 数据被索引，变成可检索资产。
4. 数据被 App 和工作流运行时消费。
5. 数据以证据、引用和追踪的形式解释给用户。
6. 数据被持续维护、评估、修复和优化。

## 产品意识矩阵

| 层级 | 用户真实问题 | 当前 Dify 能力 | Dataset 2.0 方向 | PM 关注点 |
|---|---|---|---|---|
| 来源 | 我的知识来自哪里？ | 文件、Notion、网页、外部 API | 连接式数据源、同步状态、来源新鲜度 | 数据源接入、授权、同步、失败恢复 |
| 处理 | 原始数据如何变成知识？ | 处理规则、分段、索引 | 导入/发布任务和阶段化处理 | 降低复杂度，解释成本和失败原因 |
| 检索 | 系统如何找到知识？ | 检索方式、`top_k`、重排 | Fast / Deep / Research 与证据结果 | 默认策略、非法组合、质量护栏 |
| 工作流 | 应用如何使用知识？ | Knowledge Retrieval 节点 | 强字段契约和明确输出模式 | 必填字段、运行时清晰度、下游消费 |
| 证据 | 我能信任这次回答吗？ | 命中记录和引用片段 | `EvidenceBundle`、引用、可回答状态 | 解释性、可信度、缺失证据 |
| 质量 | 我如何持续改进？ | 查询日志和命中次数 | 追踪、坏例、黄金问题、评估集 | 让 RAG 运营成为日常工作 |
| 管理 | 系统是否健康？ | 有限的状态和重试 | 健康检查、保留策略、重建、清理 | 安全的管理员控制面 |

## 高频用户关注点

构建者最常关注：

- 上传是否成功。
- 文档是否可用。
- 检索结果是否相关。
- 工作流节点是否配置正确。
- 元数据过滤是否按预期生效。
- 知识库为什么不能被工作流使用。

管理员最常关注：

- 数据源同步失败。
- 索引失败或过期。
- 外部知识 API 是否稳定。
- 成本、存储、保留周期。
- 权限边界是否清楚。
- 哪些知识库被哪些应用和工作流使用。

产品经理最应该关注：

- 用户是否能选择正确的知识库类型。
- 用户是否能理解知识库为什么可用或不可用。
- 用户是否能调试一次不好的检索结果。
- 用户是否能看清工作流实际收到什么。
- 用户是否能在不读日志的情况下提升 RAG 质量。
- 复杂能力是否被放在正确的界面层级，而不是堆到创建流程里。

## 产品设计原则

- 不要把所有能力塞进创建流程；创建流程只负责完成第一次可用。
- 把长期运营能力放进 Dataset Workbench。
- 把运行时配置放进工作流节点面板。
- 把可靠性信号放进证据和质量页。
- 把技术状态翻译成用户下一步动作。
- 在运行前阻断非法检索组合，而不是让用户在运行失败后排查。
- 对普通构建者隐藏底层存储和任务实现细节。
- 对管理员暴露健康、重建、保留、清理等可控能力。

## 决策分层

### 普通构建者

应该暴露：

- 数据源。
- 文档状态。
- 检索设置。
- 证据测试。
- 元数据。
- 使用位置。

应该隐藏：

- 对象存储 key。
- 原始索引投影 fingerprint。
- lease。
- fsck 枚举。
- GC 任务细节。

### 工作流构建者

应该暴露：

- 查询变量。
- 附件变量。
- 知识库选择。
- 检索模式。
- 检索深度。
- 元数据过滤。
- 输出变量结构。
- 非法组合提示。

### 管理员 / 运维者

应该暴露：

- 健康检查。
- 保留策略。
- 重建索引。
- 存储清理。
- 失败任务。
- 外部服务预算和限流。

### 开发者

应该通过文档或 API 暴露，而不是放进普通界面：

- manifest 结构。
- 解析产物结构。
- projection 内部结构。
- 原始 trace payload。
- 对象存储布局。

## 产品定义

Dataset 2.0 是 Dify 面向 App 与工作流运行时的知识供给工作台。它把用户数据变成可检索、可过滤、可解释、可运营的证据上下文。
